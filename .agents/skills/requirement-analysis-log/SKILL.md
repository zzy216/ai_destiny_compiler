# 需求分析记录

## 目标

为每个实施类需求创建简洁、基于证据的分析记录。让假设可见、控制需求范围，并保留“为什么这样改”的决策依据，而不只是记录修改了哪些代码。

## 工作流程

### 1. 明确需求

修改代码前，将用户请求转化为可验证的目标。

- 说明用户遇到的问题和期望结果。
- 区分明确要求与推断假设。
- 使用可观察的结果定义验收标准。
- 当需求可能被宽泛理解时，明确非目标范围。
- 记录未解决的问题；只有在不会实质改变范围或行为时，才自行做出安全假设。

### 2. 检查当前状态

在提出方案前，从代码仓库收集证据。

- 检查工作区状态和相关的近期提交。
- 定位与需求相关的入口、模块、测试、配置和文档。
- 沿现有数据流或控制流追踪到足以确定真实集成点的位置。
- 优先使用仓库证据而非猜测；记录文件路径和符号，不要大段复制代码。
- 保留用户已有的无关修改，并指出存在重叠的改动。

### 3. 分析范围和行为

描述满足验收标准所需的最小完整改动。

- 列出受影响的组件及其职责。
- 在适用时说明请求/响应、状态、数据或事件流转。
- 按需识别 API、数据库、界面、配置、兼容性、安全和运维影响。
- 覆盖校验、错误处理、空状态、边界条件和向后兼容性。
- 记录考虑过的替代方案，以及选择当前方案的原因。

### 4. 制定实施和验证计划

修改代码前，写出有顺序的实施计划。

- 指明预期修改的文件或模块。
- 为新增或变更行为补充测试，并为重要既有行为考虑回归测试。
- 按需纳入迁移、生成文件、文档或环境配置更新。
- 定义证明需求完成所需执行的命令或检查。
- 根据任务规模控制分析篇幅；小改动可以简写，但仍须说明范围和验证方式。

### 5. 实施并回填记录

完成实现后，回填同一份分析记录。

- 总结实际完成的改动，并说明与原计划的偏差。
- 记录实际执行的测试和检查及其结果。
- 列出已知限制、后续工作或有意延后的事项。
- 未实际执行的检查不得标记为通过。

## 记录位置和命名

将记录保存到项目文档目录。本项目优先使用 `docs/05.analysis/`；如果项目已有明确的分析记录或决策记录目录，则沿用既有目录。只有在需求需要记录时才创建目录。

每个需求使用一个 Markdown 文件：

`YYYY-MM-DD-<简短英文短横线slug>.md`

使用项目所在时区的日期和稳定、简短的英文 slug。同一需求的后续迭代应更新原记录，除非用户明确要求拆分为多份记录。

## 必须使用的记录模板

```markdown
# <需求标题>

- 日期：YYYY-MM-DD
- 状态：分析中 | 实施中 | 已完成 | 已阻塞
- 需求来源：<用户请求或 issue 引用>

## 一、需求说明

### 问题
<需要解决什么问题？>

### 预期结果
<完成后用户或系统应观察到什么？>

### 验收标准
- [ ] <可观察、可验证的标准>

### 非目标
- <明确不在范围内的内容；没有则填写“暂无”>

### 假设和待确认问题
- 假设：<假设内容及其安全依据>
- 待确认：<问题、负责人或解决方式>

## 二、当前状态

- 相关文件和符号：`<路径>:<符号>` — <相关原因>
- 现有行为：<基于证据的现状总结>
- 现有测试或检查：<相关覆盖情况>

## 三、功能分析

### 范围和受影响组件
- <组件> — <职责及影响>

### 行为和数据流
<说明重要的流程、状态变化或契约。只有在确实能提升理解时才添加小型图示。>

### 边界条件和失败处理
- <场景> — <预期处理方式>

### 风险和兼容性
- <风险> — <缓解措施>

### 方案决策
<说明选定方案及原因；必要时说明考虑过的替代方案。>

## 四、实施计划

1. <文件或模块改动>
2. <测试或文档改动>
3. <验证命令或检查>

## 五、实施结果

### 已完成改动
<说明实际实现内容及偏差。>

### 验证结果
- `<命令>` — 通过 | 失败 | 未执行

### 限制和后续工作
- <已知限制或后续工作；没有则填写“暂无”>
```

## 质量要求

- 清晰区分事实、假设、决策和结果。
- 尽可能将结论关联到具体仓库证据。
- 记录应便于未来维护者阅读；不要逐条叙述每一条 shell 命令。
- 不要在记录中写入密钥、令牌、凭据、个人信息或完整的敏感请求内容。
- 分析记录不能替代测试、代码评审或必要的用户澄清。
- 任务受阻时，写明确切阻塞原因，以及缺少的决策或外部依赖。
