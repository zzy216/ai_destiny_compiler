# 调整阶段分支与提交工作流

- 日期：2026-07-17
- 状态：已完成
- 需求来源：用户请求“每个阶段都开分支做，做完了就按阶段提交代码；完成后由用户指定下个阶段”。

## 一、需求说明

### 问题

现有 MVP 计划描述了阶段顺序、验收和测试，但没有规定每个阶段的 Git 分支、提交边界和阶段完成后的交接方式；此前阶段 2 的提交历史位于 `main`。

### 预期结果

计划明确规定：每个阶段从基线创建独立分支；阶段内按中文 Conventional Commit 提交；阶段验收通过后保留阶段分支并暂停，等待用户指定下一阶段；进入下一阶段时再合并上一阶段结果并创建新分支。

### 验收标准

- [ ] 计划包含阶段分支命名规则和阶段分支清单。
- [ ] 计划包含阶段内提交、测试、文档和完成验收规则。
- [ ] 计划明确阶段完成后不自动进入下一阶段，等待用户指示。
- [ ] 计划说明现有阶段 2 在 `main` 的历史情况，以及从下一阶段开始执行新流程。
- [ ] 本次计划调整以中文标准提交，工作区保持干净。

### 非目标

- 本次不创建下一阶段分支。
- 本次不移动、重写或拆分已有阶段 2 提交。
- 本次不推送远程分支、不创建 Pull Request。

### 假设和待确认问题

- 假设：阶段完成后默认保留本地分支并等待用户指示；是否推送或创建 PR 仍需用户明确要求。
- 假设：进入新阶段时，将上一阶段分支合并到 `main`，再从最新 `main` 创建新阶段分支。
- 待确认：用户指定下一阶段时，以用户明确的阶段编号和任务范围为准；若与清单不一致，先以用户指示为准并记录偏差。

## 三、功能分析

### 范围和受影响组件

- MVP 实施计划 — 增加阶段分支策略、提交规范和阶段交接协议。
- MVP 执行清单 — 增加阶段基线、分支、提交和验收状态字段。
- 分析记录 — 记录历史兼容策略和今后的执行边界。

### 行为和数据流

阶段开始 → 确认基线 → 创建 `codex/phase-{编号}-{slug}` → 在阶段分支开发和提交 → 运行测试/类型检查/构建/覆盖率 → 更新计划与 TDD 记录 → 创建阶段完成提交 → 停止并等待用户指定下一阶段。

下一阶段授权 → 将上一阶段分支合并到 `main` → 从最新 `main` 创建下一阶段分支 → 开始下一阶段。

### 边界条件和失败处理

- 工作区不干净 — 不创建阶段分支，先报告并保留用户修改。
- 阶段验收未通过 — 不标记完成、不合并到 `main`、不进入下一阶段。
- 用户指定的任务与清单不一致 — 以用户明确指示为准，并在分析记录中记录范围偏差。
- 需要远程推送或 PR — 默认不执行，必须由用户额外授权。

### 风险和兼容性

- 阶段 2 历史提交已在 `main`，回溯搬移会改变历史且没有收益；从下一阶段起执行新流程即可。
- 阶段完成保留独立分支，便于审查和回滚；合并动作延迟到下一阶段启动，避免未获授权时扩大 Git 操作范围。
- 分支名和提交类型使用稳定英文标识，提交主题使用中文，兼容常规 Git 工具和日志检索。

### 方案决策

采用“阶段分支 + 中文 Conventional Commit + 验收后暂停”的流程。阶段内允许多个证据清晰的提交（如 `test`、`feat`、`fix`、`docs`），不强制压缩历史；阶段完成时以测试、文档和清单状态作为完成证据。相比每次直接在 `main` 开发，该方案能隔离阶段风险并保留阶段边界；相比立即推送或创建 PR，暂停交接更符合用户逐阶段授权的协作方式。
