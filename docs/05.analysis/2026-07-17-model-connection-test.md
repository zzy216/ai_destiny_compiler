## 一、需求说明

### 问题

模型适配器已经可以调用 OpenAI-compatible 和 Ollama 协议，但模型管理 Controller 仍全部返回 `501 contract_not_implemented`，无法从数据库中的模型配置执行连接测试。

### 预期结果

管理员或自定义模型测试路由能够读取模型当前版本和加密凭据，调用对应适配器，并返回统一的可达性、延迟和脱敏错误摘要；响应不包含明文凭据或供应商原始错误体。

### 验收标准

- [ ] 系统模型测试路由能够按模型 ID 读取当前草稿版本（无草稿时使用已发布版本）。
- [ ] 自定义模型测试路由能够按模型 ID执行同样的连接测试，并保留后续用户归属校验接入点。
- [ ] OpenAI-compatible、Ollama 和不支持的协议分别得到正确结果或安全错误。
- [ ] 加密凭据只在调用适配器前解密，接口响应和错误不回显凭据。
- [ ] 数据库关闭时原有契约测试仍保持 `501` 占位行为。
- [ ] 新增测试先形成 RED，再实现至 GREEN；全量测试、类型检查和构建通过。

### 非目标

- 本次不实现模型 CRUD、发布/停用/默认模型规则。
- 本次不实现认证、管理员角色和用户资源归属校验；这些属于后续认证阶段。
- 本次不允许通过连接测试接口扩展任意请求方法或返回供应商原始响应体。

### 假设和待确认问题

- 假设：在认证尚未实现的阶段，测试路由暂按模型 ID执行；Service 的方法边界保留后续注入当前用户和角色校验的空间。
- 待确认：暂无。用户未提供新的模型供应商协议，因此仅接入已有两种适配器。

## 三、功能分析

### 范围和受影响组件

- `ModelsService` — 读取模型配置、选择版本、解密凭据并调用适配器。
- `ModelsModule` — 注册 TypeORM Repository 和 Service，同时兼容测试环境不启用数据库的契约测试。
- `models.controller.ts` — 将系统和自定义模型测试路由接入 Service，统一包装响应。
- 模型测试单元/集成测试 — 覆盖成功、缺少模型、缺少版本、凭据解密和不支持协议等路径。
- `.env.example` — 将误写入的固定密码改回非敏感占位符。

### 行为和数据流

请求模型 ID → 查询 `model_configs` → 选择 `currentDraftVersionId` 或 `publishedVersionId` → 查询 `model_config_versions` → 按 `protocol` 选择适配器 → 必要时查询并解密 `model_credentials` → 发起一次最小 `ping` 请求 → 返回安全连接结果。

### 边界条件和失败处理

- 数据库或 Repository 未注册 — 保留 `501`，不破坏当前契约阶段测试。
- 模型不存在、无可测试版本或版本记录缺失 — 返回明确的 404/400 错误，不发起外部请求。
- 凭据不存在 — 允许无凭据协议继续由供应商返回安全归一化结果；需要凭据的失败不暴露密钥内容。
- 适配器抛出未知异常 — 转换为安全的不可达结果，避免泄露原始错误。
- 不支持的协议 — 返回 400，不执行网络请求。

### 风险和兼容性

- 凭据泄露 — 仅把解密值传给适配器，响应和异常使用适配器的安全结果；测试和文档不写入真实密钥。
- SSRF — 本次不新增模型创建入口；沿用 DTO 对自定义模型的 HTTPS 约束，并将网络地址策略留给模型写入 Service 和安全测试阶段。
- 测试环境缺少数据库 — Repository 依赖使用可选注入，未注册时继续返回契约占位错误。

### 方案决策

采用一个 `ModelsService` 统一承载两类测试路由，而不是在 Controller 中直接操作 Repository。这样可以集中处理版本选择、凭据解密、协议分派和错误归一化，并为后续 CRUD、发布和认证授权复用。暂不实现完整模型管理，避免扩大本批次范围。
