# 实施阶段 5 自建认证

- 日期：2026-07-18
- 状态：已完成
- 需求来源：用户要求“继续下一步”，依据 MVP 清单进入阶段 5。

## 一、需求说明

### 问题

阶段 4 已交付 React 管理后台，但 API 仍大量依赖开发期固定用户和固定管理员上下文；Auth Controller 只有契约占位，生产环境不能可靠区分普通用户、管理员和资源归属。

### 预期结果

从最新 `main` 创建 `codex/phase-5-auth`，实现 NestJS 自建认证纵切：邮箱或用户名登录、邀请码注册、Access Token、Refresh Token 轮换、登出、改密、管理员鉴权、用户资源归属和关键接口限流；React 管理后台接入真实管理员登录态。

### 验收标准

- [ ] 登录和注册使用 Argon2id 校验或写入密码哈希，错误响应不暴露账号是否存在。
- [ ] 邀请码和 Refresh Token 只保存 HMAC 摘要，不保存明文。
- [ ] Refresh Token 使用后轮换，旧 Token 重用会撤销同一 Token Family。
- [ ] `Authorization: Bearer` Access Token 能识别当前用户，改密后旧 Access Token 失效。
- [ ] 普通用户只能访问自己的会话和自定义模型，管理员才能访问后台模型、教练配置、知识卡和 Agent Run。
- [ ] 登录、刷新、改密和模型消息接口有进程内限流。
- [ ] React 管理后台通过登录获取管理员会话，未登录或非管理员不能进入管理页面。
- [ ] 相关测试、类型检查、构建、覆盖率、OpenAPI 生成和文档证据通过。

### 非目标

- 不实现密码找回、验证码、邮箱验证、第三方登录、复杂权限树、审计日志和多实例共享限流。
- 不实现 Flutter 客户端页面；阶段 6 再接入移动端安全存储和登录流程。
- 不增加 Redis；限流采用进程内窗口，后续多实例部署再替换为共享存储。

### 假设和待确认问题

- 假设：Access Token 可以使用 Node `crypto` 实现 HMAC-SHA-256 签名的短期 Bearer Token，避免本阶段引入额外 JWT 依赖。
- 假设：后台登录态先使用内存中的 Access Token 调用 API，并由 Refresh Token 续期；HttpOnly Cookie 与 CSRF 可作为后续生产加固项补齐。
- 待确认：暂无；阶段验收以 API 安全边界和后台可登录运行为准。

## 三、功能分析

### 范围和受影响组件

- `apps/api/src/auth` — 认证服务、Token 签发校验、当前用户装饰器、角色守卫和限流守卫。
- `apps/api/src/models` — 自定义模型使用当前用户 ID，管理员模型使用当前管理员 ID。
- `apps/api/src/conversations` — 会话创建、查询和发消息使用当前用户 ID。
- `apps/api/src/admin` — 替换开发期守卫为真实管理员角色守卫。
- `apps/admin` — 登录页、会话上下文、API Client Authorization 头、未登录状态处理。
- `docs/04.testing`、`docs/03.plan`、`docs/05.analysis` — 阶段 5 TDD 证据、MVP 清单和分析结果。

### 行为和数据流

```text
用户登录或用邀请码注册
→ AuthService 校验账号、密码或邀请码
→ 签发短期 Access Token 和随机 Refresh Token
→ Refresh Token 摘要写入 auth_sessions
→ 业务请求通过 Bearer Token 解析当前用户
→ Controller 将 currentUser.id 传给 Service
→ Service 查询同时带资源 ID 和 user_id
```

### 边界条件和失败处理

- 账号不存在或密码错误 — 返回统一 401，不泄露账号存在性，并记录失败次数。
- 用户禁用或临时锁定 — 返回 403 或 401，不签发 Token。
- Refresh Token 过期、撤销或重用 — 拒绝刷新；重用时撤销同一 Token Family。
- 改密 — 校验旧密码，更新密码哈希和 `passwordChangedAt`，撤销既有会话。
- 非管理员访问管理端点 — 返回 403。
- 未登录访问受保护端点 — 返回 401。
- 高频登录、刷新、改密或发消息 — 返回 429，错误体不包含密码或 Token。

### 风险和兼容性

- 自建 Token 实现容易出错 — 只采用 HMAC、固定算法、过期校验、`iat` 与改密时间比较，并添加单元测试覆盖。
- 进程内限流不适合多实例 — 文档标记限制，未来接入 Redis 或网关限流。
- 前端 Token 暂存在内存会导致刷新页面后需重新登录 — 接受该 MVP 权衡，避免把 Token 写入 localStorage。
- 阶段 5 会改变原来无认证的 API 行为 — 更新契约测试和前端请求逻辑。

### 方案决策

采用“后端认证能力优先 + 前端最小登录态接入”的纵切方案。AuthService 负责密码、邀请码、Refresh Token 和 Access Token；控制器只做 DTO 与请求上下文适配。业务模块通过统一 `CurrentUser` 注入用户 ID，管理员端点统一使用角色守卫。限流先使用进程内固定窗口，满足单实例 MVP 和测试要求。
