# MVP 内测发布清单

> 适用版本：MVP 0.1  
> 阶段证据：[阶段 8 测试和发布 TDD](../04.testing/13.测试和发布.tdd.md)

## 一、发布前环境

- Node.js、pnpm、Flutter 和本机 PostgreSQL 可用。
- `apps/api/.env` 已按 `apps/api/.env.example` 配置，生产或内测环境不得使用测试密钥。
- PostgreSQL 数据库已创建，连接账号只授予应用所需权限。
- 管理后台和 Flutter 客户端的 API 地址指向同一套内测 API。

## 二、数据库检查

- 执行 `cd apps/api && pnpm migration:run`，确认 20 个 Migration 已应用。
- 首次内测环境执行 `cd apps/api && pnpm seed` 创建管理员、测试用户、默认模型占位、教练配置和知识卡。
- 发布前备份数据库；至少包含 `users`、`model_configs`、`model_config_versions`、`conversations`、`messages`、`agent_runs`、`action_cards`、`execution_records`、`reviews` 和 `memories`。
- 回滚策略优先恢复数据库备份；代码回滚到上一合并提交后再执行必要的 Migration revert。

## 三、契约和接口调试

- 执行 `cd apps/api && pnpm generate:openapi`，确认 `apps/api/openapi.json` 与当前代码一致。
- 将 `apps/api/openapi.json` 导入 Apifox 或等价接口工具，配置内测环境变量：
  - `baseUrl`：内测 API 地址。
  - `accessToken`：登录接口返回的 Access Token。
  - `refreshToken`：登录接口返回的 Refresh Token。
- 抽查登录、刷新、模型列表、创建会话、发送消息、行动卡查询、执行反馈、复盘和记忆接口。

## 四、统一验收

- 执行仓库根目录命令：`./scripts/verify-release.sh`。
- 验收脚本必须通过 API 覆盖率、API 类型检查、API 构建、OpenAPI 生成、管理后台覆盖率、管理后台类型检查、管理后台构建、Flutter 测试、Flutter 静态分析和空白差异检查。
- 如果 OpenAPI 生成后出现 `apps/api/openapi.json` 差异，先审查差异并提交，再重新执行验收脚本。

## 五、发布后冒烟

- API：访问 `/api/health` 和 `/api/docs-json`，确认服务和契约可访问。
- 认证：用管理员和普通用户分别登录，确认角色权限隔离。
- 管理后台：创建或编辑测试模型，执行连接测试，确认不会显示明文凭据。
- Flutter 客户端：登录、创建咨询、发送消息、查看历史会话和个人设置。
- 行动闭环：确认咨询生成行动卡后，可通过 API 查询行动卡、提交执行反馈、创建复盘和管理记忆。

## 六、已知限制

- 阶段 8 不执行真实远程部署、推送、PR 或应用商店发布。
- Apifox 导入需要外部工具手动完成，仓库只维护 OpenAPI 真相源。
- 复盘、记忆和提醒的 AI 自动化仍是后续增强项。
