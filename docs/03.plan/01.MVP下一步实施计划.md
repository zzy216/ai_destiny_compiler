- 模型配置如何发布和保留版本；
- 会话如何固定所选模型；
- Agent Run 如何追溯实际模型版本；
- 自建认证如何保存密码和 Refresh Token；
- 用户数据如何隔离；
- 行动卡和执行反馈如何形成闭环。

- API Key、密码和 Refresh Token 都不保存明文；
- 已经被会话引用的模型不能被物理删除；
- 表结构足以支撑一次完整的咨询、行动和复盘；
- 可以直接根据文档编写 TypeORM Entity 和 Migration。

### 3.2 不创建独立 API 文档：采用代码即契约

本项目不再维护一份手写的 `03.API接口设计.md`。API 契约直接写在 NestJS 代码中，由代码自动生成接口文档。

数据库设计确认后，在 `apps/api` 中同步实现：

- Request DTO；
- Response DTO；
- `class-validator` 输入校验；
- Swagger/OpenAPI 装饰器；
- Controller 路由和 HTTP 状态码；
- Service 权限和业务规则；
- E2E 接口测试。

代码中必须覆盖：

- 统一请求、响应和错误格式；
- 登录、注册、刷新和退出；
- 后台用户和邀请码管理；
- 后台模型 CRUD、测试、发布、停用和默认模型；
- 用户自定义模型 CRUD 和测试；
- 可用模型列表；
- 创建会话并选择模型；
- 消息发送和 SSE 流式回复；
- 行动卡、执行反馈和复盘；
- 用户记忆管理；
- 管理后台统计和 Agent Run 查询；
- 权限、限流、分页和错误码。

NestJS 启动后提供：

- Swagger UI，例如 `/api/docs`；
- OpenAPI JSON，例如 `/api/docs-json`；
- 统一的请求和响应 Schema；
- 可供 Apifox 导入的 OpenAPI 文件或 URL。

Apifox 用于接口调试、Mock、环境变量和测试集合，不作为第二份接口真相。任何接口字段修改都必须先改 DTO 和 Controller，再重新生成 OpenAPI。

OpenAI SDK 只负责调用大模型，不替代 NestJS 的业务 API 契约，也不负责用户、会话、模型配置和权限接口。

代码即契约完成标准：

- Flutter、React 和 NestJS 可以依据同一份契约并行开发；
- 每个写接口都有明确的身份和权限要求；
- 创建会话时必须明确提交模型配置；
- 查询接口不会返回密码、Token 或模型明文 Key；
- 流式接口和普通 JSON 接口的职责清晰。

不需要额外维护一份与代码重复的 Markdown API 文档。

## 4. 实施阶段和先后顺序

### 阶段 0：详细设计

预计时间：2～4 天。

按顺序完成：

1. 数据库设计文档；
2. NestJS 模块、DTO 和 Swagger 基础结构；
3. 环境变量清单；
4. 首批 Migration 顺序；
5. 第一阶段开发任务拆分。

阶段验收：数据库与 DTO 字段不存在明显冲突，可以开始编写 NestJS Service 和 Controller。

### 阶段 1：基础工程

预计时间：1～2 天。

工作内容：

- 创建 NestJS、React 和 Flutter 工程；
- 本机原生安装或连接 PostgreSQL；
- 配置 TypeORM；
- 配置 Migration 和种子数据；
- 配置 Swagger/OpenAPI 和 `/api/docs`；
- 配置统一响应、错误码和异常过滤器；
- 配置环境变量校验；
- 配置 Pino 日志；
- 创建健康检查接口；
- 创建统一错误响应。

阶段验收：三个客户端均能连接 NestJS，NestJS 能稳定连接 PostgreSQL。

### 阶段 2：模型管理底座

预计时间：3～5 天。

这是第一批核心代码。

工作内容：

- 实现模型配置和版本 Entity；
- 实现 AES-256-GCM 凭据加密组件；
- 实现 OpenAI-compatible 适配器；
- 实现 Ollama 适配器；
- 实现模型连接测试；
- 实现模型发布、停用和默认模型；
- 实现 Agent Run 基础记录；
- 使用临时管理 API 或种子数据维护模型。

阶段验收：同一个 NestJS 接口能够选择并调用 API 模型或 Ollama 模型，而且数据库和日志中没有明文 Key。

### 阶段 3：Agent 核心闭环

预计时间：4～6 天。

暂时使用固定测试用户，不先做完整注册登录。

工作内容：

- 创建会话并绑定模型版本；
- 保存用户消息；
- 读取教练配置；
- 匹配法典知识卡；
- 调用会话指定模型；
- 校验结构化诊断输出；
- 创建行动卡；
- 保存助手消息和 Agent Run；
- 通过 SSE 返回流式回复。

阶段验收：通过 API 可以完成一次“选择模型—提出问题—获得诊断—生成行动卡—保存记录”的闭环。

### 阶段 4：React 管理后台

预计时间：4～6 天。

先做后台中最影响核心能力的功能：

1. 模型列表；
2. 创建和编辑 API 模型；
3. 创建和编辑本地模型；
4. 连接测试；
5. 发布、停用和设置默认模型；
6. 教练配置；
7. 法典知识卡；
8. Agent Run 查询。

阶段验收：管理员不修改代码即可维护模型、教练配置和知识卡。

### 阶段 5：自建认证

预计时间：3～5 天。

工作内容：

- 管理员创建用户；
- 邀请码注册；
- 用户名或邮箱加密码登录；
- Argon2id 密码哈希；
- Access Token；
- Refresh Token 轮换；
- 退出和全部设备退出；
- 管理员重置密码；
- 用户和管理员角色；
- 用户资源归属校验；
- 登录和模型调用限流。

阶段验收：用户只能访问自己的数据，管理员才能维护系统模型和配置。

### 阶段 6：Flutter 咨询客户端

预计时间：5～8 天。

页面开发顺序：

1. 登录；
2. 可用模型列表；
3. 用户自定义模型；
4. 创建咨询并选择模型；
5. 流式对话；
6. 诊断结果；
7. 行动卡；
8. 历史会话；
9. 个人设置。

阶段验收：用户可以选择后台模型或自定义模型开始咨询，历史会话能够展示实际使用的模型。

### 阶段 7：行动和复盘闭环

预计时间：4～6 天。

工作内容：

- 行动卡详情；
- 完成、部分完成和未完成反馈；
- 执行记录；
- 每日复盘；
- 下一步行动建议；
- 用户记忆查看和删除；
- App 本地提醒。

阶段验收：用户能够从咨询进入行动，再根据执行结果获得下一步建议。

### 阶段 8：测试和发布

预计时间：3～5 天。

工作内容：

- 50 条 Agent 评测数据；
- 后端单元和集成测试；
- React 关键流程测试；
- Flutter 关键页面测试；
- 权限和越权测试；
- Token 轮换测试；
- API Key 不回显测试；
- 自定义模型 Base URL 的 SSRF 测试；
- PostgreSQL 备份；
- 使用 systemd 或 PM2 原生运行 NestJS；
- 发布 TestFlight 和 Android 内测包。

阶段验收：能够邀请 20～30 名真实用户持续测试。
